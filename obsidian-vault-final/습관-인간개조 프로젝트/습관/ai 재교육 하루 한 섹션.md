    탐색 → 계획 → 코딩 → 커밋 

    ### 1단계: 탐색 (정보 수집)

    Claude에게 관련 파일, 이미지, URL을 읽도록 요청하되, 일반적인 지침("로깅을 처리하는 파일을 읽어줘")이나 구체적인 파일명("logging.py를 읽어줘")을 제공하고, 아직 코드를 작성하지 말라고 명시적으로 지시합니다.

      복잡한 문제일수록 Claude가 서브에이전트를 사용해 세부사항을 확인하도록 하세요. 특히 작업 초반에 이렇게 하면 전체 맥락을 잘 유지하면서도 효율적입니다.

      예를 들어 "코드 검토 전문 에이전트로 이 코드를 분석해줘" 또는 "파일 검색 전문 에이전트로 관련 파일을 찾아줘"와 같이 요청하면, 대화 초반에도 더 정확한 분석을 얻을 수 있고 전체적인 작업 품질이 향상됩니다.

      1. https://gitingest.com/
    ### 2단계: 계획 수립

    Claude에게 특정 문제에 접근하는 방법에 대한 계획을 세우도록 요청합니다. "think"라는 단어를 사용해 확장 사고 모드를 활성화하는 것을 권장하는데, 이는 Claude에게 대안을 더 철저히 평가할 수 있는 추가 계산 시간을 제공합니다. 다음 특정 구문들은 시스템에서 증가하는 사고 예산 수준에 직접 매핑됩니다.

    "think" 키워드로 깊은 사고 모드 활성화:

    - "think" → 기본 사고
    - "think hard" → 더 깊은 사고
    - "think harder" → 매우 깊은 사고
    - "ultrathink" → 최대 깊이 사고
    각 수준은 Claude가 사용할 수 있는 사고 예산(토큰)을 점진적으로 더 많이 할당합니다.

    ### 3단계: 코딩

    Claude에게 계획에 따라 실제 코드 구현하도록 요청합니다. 이 단계에서는 솔루션의 각 부분을 구현하면서 해당 솔루션의 합리성을 명시적으로 검증하도록 요청하는 것도 좋습니다.

    ### 4단계: 커밋 & PR

    Claude에게 결과를 커밋하고 풀 리퀘스트를 생성하도록 요청합니다. 관련이 있다면 이 시점에서 Claude에게 방금 한 작업에 대한 설명과 함께 README나 변경 로그(CHANGELOG)를 업데이트하도록 하는 것도 좋은 시점입니다.

    ### ⚠️ 핵심!

    #1-#2 단계는 매우 중요합니다. 이 단계 없이는 Claude가 솔루션 코딩으로 바로 뛰어드는 경향이 있습니다. 때로는 그것이 원하는 바일 수도 있지만, Claude에게 먼저 연구하고 계획하도록 요청하면 사전에 더 깊은 사고가 필요한 문제들에 대한 성능이 크게 향상됩니다.

    ## 💡 핵심

    인간과 마찬가지로, Claude의 출력은 반복을 통해 크게 향상되는 경향이 있습니다. 첫 번째 버전은 괜찮을 수 있지만, 2-3번의 반복 후에는 일반적으로 훨씬 더 나아 보입니다. 최상의 결과를 위해 Claude가 자신의 출력을 볼 수 있는 도구를 제공하세요.

    - 1차 시도: 기본적인 구조 완성
    - 2차 시도: 세부 디자인 개선
    - 3차 시도: 거의 완벽한 결과물
    Claude가 자신이 만든 결과물을 직접 볼 수 있도록 도구를 제공하는 것이 중요합니다. 보고 수정할 수 있어야 더 좋은 결과를 만들 수 있습니다.

      # 📋 미션 내용

      ### 작업 관리 도구에 대한 흔한 오해

      대부분의 사람들은 작업 관리 도구를 사용하면 이런 일이 자동으로 일어날 것이라고 기대합니다:

      ❌ 잘못된 기대:

      - "도구가 알아서 똑똑하게 작업을 나눠줄 거야"
      - "나눈 작업을 순서대로만 실행하면 완벽한 앱이 뚝딱 나올 거야"
      - "개발 로드맵도 도구가 다 알아서 짜줄 거야"
      - "복잡한 판단은 AI가 다 해줄 거야"
      하지만 그렇지 않습니다!

      ### 작업 관리 도구의 진짜 역할

      작업 관리 도구는 말 그대로 "작업을 관리하는 도구"입니다:

      ✅ 실제 역할:

      - 커다란 작업을 작은 단위로 나누기
      - 나눈 작업을 정리하기 (A하고, B하고, C하고...)
      - 작업 진행 상황 추적하기
      - 반복적인 작업 흐름 자동화하기
      도구는 작업을 관리할 수 있도록 도와줄 뿐입니다.

      핸들링 주도권은 여전히 우리가 쥐고 있습니다!

      ### 왜 우리가 중요한가?

      ### 1. 올바른 개발 순서를 결정하는 것은 우리다

      하나의 웹 애플리케이션을 만들 때도 어떤 순서로 개발하느냐에 따라 성공할 수도 있고, 실패할 수도 있습니다.

      ❌ 잘못된 접근: 기능 우선 개발

      문제점:

      - 각 기능마다 중복 코드 발생
      - 공통 로직이 없어 유지보수 어려움
      - 나중에 수정하려면 모든 기능을 다시 고쳐야 함
      - 확장성과 재사용성 낮음
      ✅ 올바른 접근: 아키텍처 우선 개발

      장점:

      - 견고한 기반 위에 체계적으로 구축
      - 코드 재사용성 극대화
      - 유지보수와 확장이 쉬움
      - 일관된 구조로 협업 용이
      소프트웨어 아키텍처 전문가 Martin Fowler는 이렇게 말합니다:

        "아키텍처는 시스템을 형성하는 중요한 설계 결정을 나타내며, 여기서 중요하다는 것은 변경 비용으로 측정됩니다."

        출처: Martin Fowler's Software Architecture Guide

      프로젝트 관리 연구에서도 작업의 순서화가 얼마나 중요한지 강조합니다:

        "프로젝트 계획 수립은 단순히 완료해야 할 작업을 나열하는 것이 아닙니다. 신중한 활동 정의와 순서화, 자원 추정이 필요합니다."

        출처: PMI - Project Planning Techniques

      ### 2. 프로젝트의 컨텍스트를 아는 것은 우리뿐이다

      작업 관리 도구는 다음을 알 수 없습니다:

      - 팀의 기술 수준과 경험
      - 프로젝트의 비즈니스 우선순위
      - 기술 부채와 레거시 시스템
      - 조직의 제약사항과 정치적 상황
      - 실제 사용자의 니즈와 요구사항
        "이해관계자가 누구인지 결정하려면 프로젝트 결과로 이익을 얻는 사람뿐만 아니라 영향을 받을 사람, 자원을 제공하는 사람, 의사 결정권을 가진 사람을 인식해야 합니다."

        출처: HBR Guide to Project Management

      예시:

      - 도구: "로그인 기능을 먼저 만드세요"
      - 우리: "아니야, 우리 팀은 백엔드가 약하니까 API 구조부터 잡아야 해"
      - 도구: "UI부터 만드세요"
      - 우리: "아니야, 레거시 시스템과 연동이 필요하니까 DB 설계부터 해야 해"
      ### 3. 지속적으로 검토하고 방향을 조정하는 것도 우리다

      프로젝트는 계획대로만 진행되지 않습니다. 실행 중에도 끊임없이 판단하고 조정해야 합니다:

      우리가 해야 하는 판단:

      - 현재 개발 순서가 올바른가?
      - 우선순위를 바꿔야 하는가?
      - 계획을 수정해야 하는가?
      - 어떤 부분을 먼저 리팩토링해야 하는가?
      소프트웨어 개발의 고전 『The Pragmatic Programmer』는 이렇게 강조합니다:

        "당신의 기술에 관심을 가지세요. 그것을 잘 하는 것에 관심이 없다면 왜 소프트웨어 개발에 인생을 바치나요? 생각하세요! 당신의 작업에 대해."

        출처: The Pragmatic Programmer

      핸들링 주도권은 항상 우리에게 있습니다!

      우리가 반드시 해야 할 일:

        - 애플리케이션 골격부터 만들기
        - 공통 모듈/컴포넌트 먼저 정의하기
        - 개별 기능은 공통 기반 위에 구축하기
        - 무엇을 먼저 만들어야 하는가?
        - 어떤 순서로 진행해야 하는가?
        - 각 단계의 목표는 무엇인가?
        - 현재 순서가 올바른가?
        - 방향을 수정해야 하는가?
        - 우선순위를 바꿔야 하는가?
